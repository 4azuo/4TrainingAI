4. Nguyên tắc giao việc cho Sub-Agent

Sub-agent có thể gọi sub-agent khác khi thực sự cần — nhưng phải kiểm soát chặt để tránh đệ quy không kiểm soát: chạy mãi không dừng, đốt token.

✅ Nên

  • Sub-agent làm một việc cụ thể rồi báo cáo kết quả về nơi giao việc
  • Được gọi sub-agent con khi cần, nhưng giới hạn độ sâu (vd tối đa 1–2 cấp) và không tạo vòng
  • Khi gặp quyết định lớn / mơ hồ → dừng và hỏi user
  • Có điều kiện dừng rõ ràng (done / fail / cần input)

❌ Tránh

  • Đệ quy không giới hạn độ sâu — A gọi B gọi C gọi... vô tận
  • Tạo vòng: A → B → A, hoặc gọi ngược Orchestrator rồi quay lại chính nó
  • Tự ý quyết định việc lớn thay vì hỏi user

5. Viết Skills & Tự động hóa bằng Script

Một skill tốt = SKILL.md (frontmatter name + description rõ ràng) + tài nguyên đi kèm (examples, templates, script).

✍️ Kinh nghiệm viết Skill

  • Description quyết định skill có được kích hoạt đúng lúc không — mô tả rõ "khi nào dùng", thêm từ khóa gợi nhớ
  • Mỗi skill một việc, đừng nhồi nhiều mục đích
  • Kèm ví dụ mẫu (good/bad) để output nhất quán
  • Giữ SKILL.md ngắn; đẩy chi tiết dài vào file phụ, chỉ nạp khi cần

⚙️ Script > "AI tự nghĩ"

  • Việc lặp lại & xác định (deterministic) → viết script (Python/Bash) để agent CHẠY
  • Tiết kiệm token, chạy nhanh, kết quả ổn định mỗi lần
  • Dành "suy nghĩ" của agent cho việc cần phán đoán, không cho việc máy móc
  • Script còn test được và tái dùng — agent thì không đảm bảo
❌ Mỗi lần lại nhờ agent "đọc 200 file, lọc, đếm, tính tổng rồi format bảng"
   → tốn token khủng, chậm, mỗi lần ra kết quả hơi khác nhau

✅ Viết 1 script: scripts/build_report.py — agent chỉ gọi: python build_report.py
   → rẻ, nhanh, kết quả y hệt nhau, dễ kiểm thử
🧠 Quy tắc vàng
Xác định được→ Viết script (logic cố định, có input/output rõ).
Cần phán đoán→ Để agent xử lý (hiểu ngữ cảnh, ra quyết định, viết nội dung).

Cả Orchestrator và Sub-agent đều cần "nhớ". Chọn định dạng theo nhu cầu — đừng nhảy thẳng lên DB khi chưa cần.

9. Công cụ hữu ích: Live Preview (Microsoft)

Extension Live Preview của Microsoft cho VSCode — chạy server xem trước HTML ngay trong editor (không cần mở trình duyệt ngoài).

👁️ Xem trước trong editor

  • Render HTML real-time, tự reload khi sửa file
  • Chạy qua server cục bộ → fetch hoạt động (đúng vấn đề file:// của tài liệu này)

🤝 Chia sẻ góc nhìn với AI

  • Tích hợp với GitHub Copilot / Codex / Claude
  • Cho phép AI "thấy" trang đã render (giao diện thực) chứ không chỉ đọc code
  • AI hiểu UI hiển thị ra sao → sửa layout/style chính xác hơn
💡 Vì sao hữu ích cho dự án AI
Vòng lặp nhanhSửa → thấy ngay → nhờ AI chỉnh tiếp, không phải mô tả bằng lời "nút bị lệch".
Đúng bối cảnhAI nhìn được kết quả render thật → giảm đoán mò, ít vòng sửa lại.
Tiện ngay đâyMở chính trang này bằng Live Preview là chạy được luôn (nạp tab động qua fetch).

11. Kinh nghiệm dùng MCP

MCP (Model Context Protocol) là một chuẩn mở để "cắm" thêm tool & nguồn dữ liệu bên ngoài vào AI — giống ổ cắm USB-C: một chuẩn, nhiều thiết bị. Một MCP server (filesystem, git, GitHub, DB...) cắm vào là AI có thêm vài tool gọi được ngay.

🔍 Phân biệt nhanh: MCP vs Tool vs RAG
🛠️ ToolMột hành động AI gọi được (gửi mail, query DB). "AI làm được gì?"
🔌 MCPỔ cắm chuẩn để mang tool/data từ ngoài vào — không phải hành động khác, mà là cách lắp tool mới. Một server = thêm vài tool.
📚 RAGTra cứu tài liệu rồi nhồi vào ngữ cảnh — chỉ đọc, không hành động. "AI biết dựa trên data nào?"
ChốtTool/MCP nói về LÀM · RAG nói về BIẾT. MCP là ổ cắm, Tool là thứ được cắm vào.

Dưới đây là các đúc kết khi đưa MCP vào dự án thật:

✅ Nên

  • Chỉ dùng khi server thật sự cần — mỗi MCP server nạp thêm tool vào ngữ cảnh, càng nhiều càng tốn token & dễ chọn nhầm tool
  • Ưu tiên server chính thức/uy tín (filesystem, git, GitHub, DB...) hơn server lạ
  • Chạy read-only khi chỉ cần đọc (vd DB) — giảm rủi ro AI ghi/xóa nhầm
  • Đặt cấu hình ở scope đúng: dự án (.mcp.json share cho team) vs cá nhân (user scope)

❌ Tránh

  • Bật cả chục server "cho chắc" → menu tool phình to, AI lẫn lộn, chậm
  • Nhét secret/token thẳng vào file cấu hình commit lên git — dùng biến môi trường
  • Cấp quyền ghi rộng (toàn ổ đĩa, DB production) khi không cần
  • Tin tưởng mù quáng server bên thứ ba — nó thấy được dữ liệu bạn gửi qua
🧭 Khi nào đáng thêm MCP?
ĐángKhi AI cần thao tác lặp lại với một hệ thống ngoài (GitHub, DB, Jira, hệ thống file...) — MCP biến nó thành tool gọi được, thay vì copy-paste thủ công.
Chưa cầnViệc một lần, hoặc đã có sẵn CLI/script đơn giản → cứ để agent chạy lệnh, khỏi dựng MCP cho nặng.
Bảo mậtCoi mỗi MCP server như một "nhân viên có chìa khóa" — chỉ trao đúng quyền tối thiểu & chọn nguồn tin cậy.
