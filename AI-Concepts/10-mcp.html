🔌 MCP — Model Context Protocol

MCP là "ổ cắm chuẩn" để nối AI với thế giới bên ngoài — một giao thức mở (open standard) giúp AI cắm thêm tool & nguồn dữ liệu mà không cần code riêng cho từng dịch vụ. Giống cổng USB-C: một chuẩn, vô số thiết bị.

1. MCP là gì?

Trước MCP: mỗi khi muốn AI dùng một dịch vụ (GitHub, DB, Slack...) phải viết tích hợp riêng — tốn công, mỗi chỗ một kiểu. MCP đặt ra một ngôn ngữ chung: bất kỳ "MCP server" nào tuân theo chuẩn đều cắm vào được, AI tự khám phá ra các tool mà server cung cấp và gọi được ngay.

❌ Trước MCP (M×N)

M ứng dụng AI × N dịch vụ = M×N tích hợp riêng lẻ phải tự viết & bảo trì.

✅ Có MCP (M+N)

Mỗi AI nói "tiếng MCP", mỗi dịch vụ có 1 MCP server. Cắm là chạy — chỉ còn M+N.

🔍 Phân biệt nhanh: MCP vs Tool vs RAG
🛠️ ToolMột hành động AI gọi được (gửi mail, query DB). "AI làm được gì?"
🔌 MCPỔ cắm chuẩn để mang tool/data từ ngoài vào — không phải một hành động khác, mà là cách lắp tool mới. Một server = thêm vài tool.
📚 RAGTra cứu tài liệu rồi nhồi vào ngữ cảnh — chỉ đọc, không hành động. "AI biết dựa trên data nào?"
ChốtTool/MCP nói về LÀM · RAG nói về BIẾT. MCP là ổ cắm, Tool là thứ được cắm vào.

3. Cách MCP hoạt động

sequenceDiagram participant AI as 🧠 AI (Host) participant C as 🔗 Client participant S as 🔌 MCP Server (GitHub) participant G as 🌐 GitHub API AI->>C: Kết nối tới server C->>S: initialize (bắt tay, hỏi khả năng) S-->>C: Danh sách tools: create_issue, list_prs... Note over AI,C: AI giờ "biết" có các tool này AI->>C: Cần tạo issue "Bug login" C->>S: tools/call create_issue(title, body) S->>G: POST /repos/.../issues G-->>S: Issue #42 created ✅ S-->>C: Kết quả C-->>AI: Đã tạo issue #42

Điểm mấu chốt: AI không cần biết trước server có tool gì. Lúc kết nối, nó hỏi server ("discovery") rồi mới biết gọi được những gì — nên cắm thêm server là tự động có thêm tool.

4. Transport: chạy Local hay Remote?

MCP server kết nối với client qua 2 kiểu "đường truyền" chính:

KiểuChạy ở đâuCách nốiHợp với
💻 Local (stdio)Tiến trình chạy ngay trên máy bạnHost khởi chạy server, nói chuyện qua stdin/stdoutTruy cập file, git, công cụ cá nhân — dữ liệu không rời máy
🌐 Remote (HTTP / SSE)Server đặt trên cloud / mạngKết nối qua URL, thường kèm xác thực (OAuth/token)SaaS dùng chung cho cả team (GitHub, Linear, Sentry...)
💡 Local hay Remote?
LocalRiêng tư, nhanh, không cần mạng — nhưng mỗi người tự cài, tự chạy.
RemoteCài 1 lần dùng chung, luôn cập nhật — nhưng dữ liệu đi qua mạng, phải lo xác thực & tin cậy nhà cung cấp.

6. Nên / Tránh khi dùng MCP

✅ Nên

  • Chỉ bật server thật sự cần — mỗi server nạp thêm tool vào ngữ cảnh, càng nhiều càng tốn token & dễ chọn nhầm tool
  • Ưu tiên server chính thức/uy tín hơn server lạ — nhất là server remote (thấy hết dữ liệu đi qua)
  • Chạy read-only khi chỉ cần đọc (vd DB) — giảm rủi ro ghi/xóa nhầm
  • Đặt cấu hình đúng scope: dự án (.mcp.json share cho team) vs cá nhân (user scope)

❌ Tránh

  • Bật cả chục server "cho chắc" → menu tool phình to, AI lẫn lộn, chậm
  • Nhét secret/token thẳng vào file cấu hình commit lên git — dùng biến môi trường
  • Cấp quyền ghi rộng (toàn ổ đĩa, DB production) khi không cần
  • Tin mù quáng một server bên thứ ba — toàn bộ dữ liệu đi xuyên qua nó
